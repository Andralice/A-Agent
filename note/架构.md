# A-Agent 项目架构说明

本文描述当前仓库的后端结构、模块边界与运行时行为，便于新人上手与二次开发。细节接口仍以 [API接口总表.md](./API接口总表.md) 为准。

---

## 1. 概述

**A-Agent（novel-agent）** 是基于 **Spring Boot 3.2**、**Java 17** 的单体应用：对外提供 **REST（小说管理 / 文风入口 / 认证）**、**QQ（NapCat）Webhook**，内部通过 **Spring AI OpenAI** 调用兼容 OpenAI 协议的对话模型，将生成结果持久化到 **MySQL（JPA/Hibernate）**，并辅以进程内锁与数据库任务行实现**跨请求的写入互斥与恢复续跑**。

---

## 2. 技术栈

| 类别 | 选型 |
|------|------|
| 运行时 | Java 17、Spring Boot 3.2 |
| Web | spring-boot-starter-web、Spring Security（JWT 可选） |
| AI | spring-ai-openai（OpenAI 兼容 Endpoint + API Key） |
| 持久化 | Spring Data JPA、MySQL、Hibernate `ddl-auto`（默认多为 update） |
| 异步 | `CompletableFuture`、自建线程池（QQ 指令）、任务表 + `CompletableFuture.runAsync`（章节批量） |
| 调度 | `@EnableScheduling`（如定时清理）；`pom.xml` 中声明 Quartz / Redis，**当前业务代码未检索到直接使用**，可视作预留或未启用 |

---

## 3. 源码分层与包结构

逻辑分层遵循：**Controller → Service（领域编排）→ Repository（JPA）→ Entity**；**Agent** 封装对大模型的单次/多轮调用；**QQ** 包负责解析与异步派发；**config / security** 负责启动行为与安全过滤器。

```
src/main/java/com/start/agent/
├── AgentApplication.java          # 入口：@SpringBootApplication、@EnableScheduling
├── Controller/                    # REST：小说、文风、认证、QQ 回调等
├── service/                       # 领域服务（IDE/磁盘上目录名可能出现 Service 大小写差异，包名均为 service）
├── repository/                    # JpaRepository
├── model/                         # JPA 实体与枚举（Novel、Chapter、GenerationTask…）
├── agent/                         # NovelGenerationAgent、润色/一致性审查等 LLM 封装
├── prompt/                        # 叙事类 Prompt 拼装（如 NarrativeCraftPrompts）
├── qq/                            # QqCommandParser、Dispatcher、指令枚举、群访问
├── config/                        # AI、任务恢复、WebSocket、库连接重试、冷启动清理等
├── security/                      # SecurityConfig、JwtService、JwtAuthenticationFilter
├── exception/                     # GlobalExceptionHandler、NotFoundException
├── dto/                           # ApiResponse 等
└── engine/                        # 世界模拟/角色引擎（WorldSimulator、CharacterAgent 等），与主线「写小说」相对独立，可按实验模块理解
```

**静态与运维**：`src/main/resources/application.yml`（及 `application-local.yml` 等 profile）、仓库根 `pom.xml`（artifact：`novel-agent`）。

## 4. 对外入口

### 4.1 HTTP API

- **`/api/novel/**`**：`NovelManagementController`、`NovelOutlineConfigController` — 小说 CRUD、大纲、章节、进度、导出、生成任务、删除守卫等。
- **`/api/novel-style/**`**：`NovelStyleController` — 按文风创建/续写等封装入口。
- **`/api/auth/**`**：`AuthController` — 登录与当前会话；与 `AppSecurityProperties`（`app.security.*`）联动。
- **全局异常**：`GlobalExceptionHandler` 统一 JSON 错误形态（与前端约定的 `code` / `message` 等）。

认证开启时：**JWT 过滤器**解析 Bearer；业务侧通过 **`NovelLibraryAccessService`** 等对「仅管理员可见」的书做 **404** 隐藏（避免泄露资源存在性），详见 API 文档。

### 4.2 QQ / NapCat

- **`QQMessageController`** 接收 POST（原始 payload）。
- **`QqCommandParser`** → **`QqCommand`** → **`QqCommandDispatcher`** 异步调用 **`NovelAgentService`**（写书、续写、列表、读章、自动续写等）。
- **`NapCatMessageService`** + **`WebSocketConfig`**：与 NapCat 的 WebSocket 配置与发消息能力（具体 URL/token 见配置）。

## 5. 核心领域服务（职责速览）

| 服务 | 职责摘要 |
|------|-----------|
| **NovelAgentService** | 小说创作主编排：开书、续写、自动续写、章节重生、大纲重生、bootstrap、与 QQ 推送格式化；协调 Agent、仓库、进度与守卫锁 |
| **NovelGenerationAgent** | 大纲、角色、章节正文等 **LLM 调用**与结构化解析（含流水线 `WritingPipeline`、文风提示） |
| **GenerationTaskService** | **`generation_task` 表**：入队、claim、心跳、完成/失败、区间重叠校验、**大纲租约 `OUTLINE_REGENERATE`** 与章节类任务互斥 |
| **RegenerationTaskGuardService** | **进程内**章节区间锁 + 大纲重写锁，与 DB 任务锁配合防并发踩章节 |
| **WritingProgressService** | 持久化「当前写作阶段 / 区间 / 游标」，供监控接口聚合展示 |
| **CharacterProfileService** / **NewCharacterIngestService** | 角色档案生成、入库与新角色摄取 |
| **ChapterFactService** / **StoryMemoryService** / **PlotSnapshotService** | 章节事实、叙事记忆、阶段性主线快照 |
| **EntityConsistencyService** / **ConsistencyAlertService** | 实体一致性检测与告警记录 |
| **PolishingAgent** / **ContentReviewAgent** / **ConsistencyReviewAgent** | 润色与审查类 LLM 链路（按业务流程接入） |
| **GenerationLogService** | 生成日志落库 |
| **NovelExportService** / **NovelDeletionService** | 导出文本、带确认条件的硬删除 |
| **NovelLibraryAccessService** | 书库可见性与管理员守卫 |
| **CleanupService** | 定时任务（如空书清理） |
| **UserStatisticsService** | 用户/群组统计 |

## 6. 数据模型（持久化概要）

主要实体与仓库对应关系：

- **Novel**：题材、设定、大纲正文（`description`）、流水线、文风 JSON、书库标志等。
- **Chapter**：章节正文与序号。
- **CharacterProfile**、**ChapterFact**、**PlotSnapshot**、**ConsistencyAlert**：设定与连续性相关。
- **GenerationTask**：异步章节任务状态机（`PENDING` / `RUNNING` / …）、区间、`taskType`（含 **`OUTLINE_REGENERATE` 占位租约**）。
- **GenerationLog**：一次生成动作的日志。
- **UserStatistics**：统计字段。

表结构由实体 + Hibernate 演进维护；生产环境建议有独立迁移策略（当前项目以 `ddl-auto` 为主时需自行评估）。

## 7. 并发、互斥与恢复

1. **DB 层**：`GenerationTaskRepository` 等对重叠区间计数；章节类任务入队前检查；**大纲重写**插入 `OUTLINE_REGENERATE` 与章节类任务双向互斥。
2. **进程内**：`RegenerationTaskGuardService` 维护每小说的区间列表与大纲锁；**先大纲内存锁，再 DB 租约**，避免窗口期。
3. **恢复**：`GenerationTaskRecoveryRunner` 在 **`ApplicationReadyEvent`** 后调用 `GenerationTaskService.resumeRecoverableTasks()`，恢复 `PENDING`/`RUNNING`；孤儿 **`OUTLINE_REGENERATE`** 由 worker 路径收尾为结束态，避免永久阻塞（见代码与 API 文档说明）。

## 10. 演进注意点

- **单体边界清晰**：新增能力优先落在明确 Service + 可选新 Agent；避免在 Controller 内堆叠长逻辑。
- **长耗时操作**：大纲重生等为同步 HTTP，线程占用时间长；异步章节写入依赖任务表，扩展为多实例时需共享同一 MySQL 并依赖现有 DB 锁。
- **依赖冗余**：`pom.xml` 中 Redis、Quartz 若长期未使用，可考虑移除或补充实际用法，降低部署疑惑。
