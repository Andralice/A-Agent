# 角色三层注入与状态闭环——完整实施计划

本文档面向 **A-Agent** 现有架构（`CharacterProfile`、`chapter_fact`、`plot_snapshot`、`narrative_state_json` / M9、章节生成流水线），给出从「单一大档案注入」迈向 **静态卡 + 动态状态 + 本章子集注入 + 写后状态更新** 的完整路线图。目标：**可控长篇**——减轻台词模板化、人设冻结与提示过载。

---

## 一、目标与原则

### 1.1 要解决什么问题

- **静态过载**：全书角色档案整块塞进上下文 → 模型忽略关键约束、多人同声。
- **人设不演进**：只有出厂设定，缺少「本章该怎么演」的态度与短期记忆 → NPC 感。
- **缺少闭环**：正文生成后没有把「关系/态度/记忆碎片」结构化写回 → 跨章难以累积一致性。

### 1.2 设计原则（须贯穿实现）

1. **角色不是数据库副本**：入库的是「锚点 + 状态」，不是剧本全文；正文仍以大纲与章节任务为准。
2. **三层分离**：静态（少变） / 动态（常变） / 本章注入（极短、只带需要的侧面）。
3. **防信息过载**：本章显性登场角色建议上限 **3～5**（特殊群像章可配置放宽）；静态卡字段控制在 **6～8 个短字段**量级。
4. **动态覆盖静态**：注入时明确优先级：**本章 focus + 动态状态** 高于笼统静态描写。
5. **与现有能力共存**：不推翻 M4/M7/M9/sidecar/`character_state` 事实链；新能力优先 **聚合或扩展**，避免平行数据源打架。

## 二、概念模型（与参考对齐）

### 2.1 静态卡（Character Core）

- **含义**：出厂锚点，短、尖、可辨识。
- **建议字段示例**（每项一两行，禁止长篇小传）：`id`、`name`、`role`、`traits[]`（≤3）、`speechStyle`（可与现有 `voice` 合并或互映射）、`perceptionFocus`、`taboo[]`、`hook`。
- **存储取向**：可与现有 `CharacterProfile.profile_content` 渐进共存——阶段 1 仍以文本为主；阶段 2 增加 JSON 列或约定 `profile_content` 内嵌 `core` 块并由服务端解析。

### 2.2 动态状态（Character State）

- **含义**：随剧情变化，直接影响「这一章怎么演」。
- **建议字段示例**：`status`、`location`（或抽象区位）、`attitudeToProtagonist`、`recentMemory[]`（短句数组）、`currentGoal`、`stressLevel` 或离散 `currentEmotion`、`lastChapterRef`（可选）。
- **存储取向**：优先 **书本级或角色级 JSON**（新表或扩展现有实体），章节结束时 **PATCH**；或把增量写入 **`chapter_fact`**（`fact_type` 扩展或结构化 `payload`），由聚合服务合并为「当前 effective state」。

### 2.3 本章注入（Chapter Cast）

- **含义**：**不是**把所有人塞进 prompt，而是 **登场 id 列表 + 每角色本章 focus**（标签数组即可）。
- **示例语义**：「本章艾琳：警惕、试探」「本章莉莉丝：适应环境、轻微吐槽」。
- **注入位置**：初稿（及必要时润色前摘要）拼接 **独立小块**，置于角色档案之前或之后（需在实验中固定顺序，避免被大纲淹没）。

### 2.4 写后状态更新（State Delta）

- **含义**：章节成功落库后，一次 **低温、结构化** 调用（或规则 + 模型混合），输出对本章涉及角色的 **增量**：如 `attitude` 变化、`recentMemory_add[]`、`stressLevel` 等。
- **与现有关系**：可与 **sidecar / 章节叙事产物** 并列；或与 **`chapter_fact` 写入** 合并为统一 pipeline 的一步，避免两处各写各的。

## 三、与当前仓库的映射

| 能力 | 现状 | 本计划中的位置 |
|------|------|----------------|
| 角色档案 | `CharacterProfile` + JSON strict 解析 + `voice` / 对白声纹 | 静态卡载体；逐步结构化 Core |
| 事实记忆 | `chapter_fact`、`character_state` 等 | 动态状态来源之一或 delta 落点 |
| 跨章叙事 | M4 承接、`narrative_state_json`（M9） | 聚合层可吸收「effective character state」摘要 |
| 章节生成 | `NovelGenerationAgent.generateDraft` + `NovelAgentService.gen` | 插入「本章 Cast + focus」与「动态状态摘录」 |
| 配置与运维 | `application.yml`、`writingStyleParams` | 开关：是否启用 cast 裁剪、每章人数上限、是否强制 state delta |

## 四、数据与 API 设计（建议）

### 4.1 数据存储（分阶段）

**阶段 A（最低侵入）**

- 本书新增可选 TEXT/JSON：`novel.character_cast_config_json` 或复用 `writingStyleParams` 根下扩展键（需与前端/API 校验约定）。
- 本章参数：沿用现有「本章设定」字段（若已有 `chapterSetting` / 类似入口），约定一段 **结构化 JSON 文本**（服务端校验失败则降级为纯文本附录）。

**阶段 B（推荐）**

- 新表 `novel_character_state`：`novel_id`、`character_key`、`state_json`、`updated_at`、`source_chapter`。
- 或 `character_profile` 增列 `core_json`、`state_json`，由迁移脚本渐进填充。

**阶段 C（闭环加强）**

- 新表 `chapter_character_delta`：`novel_id`、`chapter_number`、`delta_json`、`created_at`，便于审计与回放。

### 4.2 API（示意）

- `GET /api/novel/{id}/character-states`：只读聚合（用于导演台调试）。
- `PATCH /api/novel/{id}/character-state/{key}`：手工修正（管理员）。
- 生成路径：**内部服务方法** 优先于对外 REST，减少暴露面；对外仅暴露「本书写作参数」合并读写若已有惯例。

（具体路径与鉴权需与 `API接口总表.md` 修订同步。）

## 五、流水线改造步骤（实现顺序）

### 阶段 0：基准与开关（1～2 天）

- 增加功能开关：`novel.generation.character-cast-enabled`、`character-cast-max-names`、`character-state-inject-enabled`、`character-state-delta-enabled`（默认值保守，避免破坏旧书）。
- 日志：本章注入块字符数、登场人数、是否应用 delta。

### 阶段 1：纯 Prompt 层「本章 Cast + Focus」（不强制落库）（3～5 天）

- 输入来源优先级：本章请求体 > 书本默认 > 无（保持现状）。
- 在 `NovelAgentService` / `generateDraft` 前拼接：
  - `【本章登场与表现焦点】` + JSON 或固定格式文本；
  - 明确要求：**未在列表中的核心角色仅允许侧写提及，不得抢戏**（可配置软硬）。
- 验收：同书固定大纲下，对比开关前后 **登场人数与台词同质化**（抽样人工 + 简单启发式统计）。

### 阶段 2：动态状态「只读注入」（5～10 天）

- 从 `novel_character_state` 或聚合查询读取 **本章登场角色** 的 state 摘录（带长度上限）。
- 与 M9 快照拼接策略：**去重合一**，避免同一事实两块重复；在 `NovelAgentService` 内单一方法组装「叙事上下文包」。
- 验收：跨章态度连续；Regression：旧书无 state 时行为与阶段 0 一致。

### 阶段 3：写后 State Delta（核心闭环）（10～20 天）

- 在章节 **成功落库且通过最小校验** 后触发：
  - 输入：本章正文摘要（或 sidecar 已有摘要）、本章 cast、写前 state、大纲一句；
  - 输出：严格 JSON schema 的 `delta`；
  - 合并策略：浅合并数组追加、标量覆盖、冲突记录日志。
- 失败策略：delta 解析失败则 **跳过写回**，不影响章节已成功事实。
- 验收：连续生成 N 章，`attitude` / `recentMemory` 与正文粗一致；人工抽检串味率。

### 阶段 4：静态 Core 结构化（与档案重生联动）（并行可穿插）

- 角色生成 prompt / 解析：`core` 字段与现有 `voice`、`want/fear` 并存映射。
- **重生角色档案** 任务：可选「仅补 `core`+`voice`，不写长传记」。
- 验收：单角色档案字符数下降，注入块反而更可执行。

### 阶段 5：检索式记忆（RAG）（可选，中长期）

- 对 `chapter_fact`、关键对白、状态变更建简易向量或关键词索引；**本章注入前检索 Top-K**。
- 仅当阶段 2～3 稳定后加入，避免过早复杂化。

## 六、风险与缓解

| 风险 | 缓解 |
|------|------|
| Delta 幻觉（正文未写却改状态） | 低温 + 短输出；强制「仅允许引用本章摘要中的事实」；可选二次校验调用 |
| 注入仍过长 | cast 人数硬上限；state 每角色最多 N 条 memory；总字符 budget |
| 与 M4/M9 重复 | 单一「上下文组装器」、字段级去重规则文档化 |
| 旧书兼容 | 全开关默认关；无 state 走旧路径 |

---

## 七、验收清单（里程碑）

1. **功能**：开关全关时与当前线上行为等价（自动化 smoke）。
2. **Cast**：开启后本章 prompt 出现独立注入块，且登场列表可追踪日志。
3. **State 注入**：开启后 prompt 含动态摘录，旧书无崩溃。
4. **Delta**：成功解析率与回滚安全；DB 可查询最新 state。
5. **体验**：样板书 10～20 章人工阅读：人设演进可感知、战术台词重复率下降（结合既有 `voice` 与反模板提示词）。

## 九、一句话总结

**方向**：静态卡锚定口吻与人格底线，动态状态驱动「这一章怎么演」，本章 Cast 控制注入与焦点，写后 Delta 闭环演进——在现有事实链与 M9 之上增量接入，用开关与向后兼容控制范围。

## 十、实施记录（仓库已实现）

- **表**：`novel_character_state`（`ddl-auto: update` 自动建表）。
- **配置**：`novel.generation.character-cast-enabled` / `character-state-inject-enabled` / `character-state-delta-enabled` 等（默认 `false`，与旧行为一致）。
- **本章设定 JSON**：`chapterSetting` 可为 `{"notes":"…","chapterCast":{"characters":[{"name":"…","focus":["…"]}],"restrictOthersToBackground":true}}`；纯文本仍兼容。
- **全书默认**：`writingStyleParams.chapterCastDefault`（结构同 `chapterCast`）；`WritingStyleParamsSupport` 已识别该键以便单独保存。
- **流水**：`NovelAgentService.genInner` → `CharacterNarrativeStateService.resolveForChapter` → 初稿 prompt 在大纲与档案之间插入调度块；落库后 `maybeApplyStateDelta`。
- **API**：`GET /api/novel/{novelId}/character-narrative-states`；`GET .../config/frontend-runtime` 增加 `characterCastEnabled`、`characterStateInjectEnabled`、`characterStateDeltaEnabled`。
